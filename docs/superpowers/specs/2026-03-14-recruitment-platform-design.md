**版本**: v1.0
**日期**: 2026-03-14

---

## 1. 项目概述

### 1.1 背景

企业招聘过程中，简历来源分散（多招聘平台 + 手动上传），HR 筛选简历耗时耗力。需要构建一个 SaaS 化的招聘平台，实现：

- 多平台简历统一管理（Boss 直聘对接 + 手动上传）
- AI 智能匹配（输入 JD 自动推荐最匹配简历）
- 多部门共享 + 权限管理

## 2. 功能模块

### 2.1 用户与权限模块

#### 角色设计

| 角色 | 权限范围 |
|------|----------|
| 超级管理员 | 系统管理，跨企业数据访问 |
| HR 管理员 | 本企业全部权限 |
| 部门管理员 | 本部门数据管理 |
| 普通用户 | 查看、匹配操作 |

#### 部门与用户关系

```
User ←→ Department (多对一)
User ←→ Role (多对多)
Role ←→ Permission (多对多)
```

#### 认证机制

- JWT Token
- 访问令牌 + 刷新令牌
- Token 有效期：访问 2 小时，刷新 7 天

---

### 2.2 简历管理模块

#### 简历来源

| 来源 | 说明 |
|------|------|
| Boss 直聘 API | 企业授权后自动同步 |
| 手动上传 | PDF/Word/图片格式 |

#### 简历解析流程

```
上传简历 → 文件存储(MinIO/OSS) → 文本提取 → AI 结构化提取 → 入库
```

#### 结构化字段

| 类别 | 字段 |
|------|------|
| 基本信息 | 姓名、手机、邮箱 |
| 教育背景 | 学历、毕业院校、专业、毕业时间 |
| 工作经历 | 公司、职位、入职/离职时间、工作职责 |
| 项目经验 | 项目名称、角色、技术栈、项目描述 |
| 技能 | 技能标签（数组） |

#### 搜索能力

- PostgreSQL 全文搜索
- 多条件过滤（学历、工作年限、技能）

---

### 2.3 JD 管理模块

#### 功能

| 功能 | 说明 |
|------|------|
| CRUD | 职位描述增删改查 |
| 状态 | 草稿、发布中、已暂停、已结束 |
| 关联 | JD 与简历的匹配记录 |

#### 结构化字段

| 字段 | 类型 |
|------|------|
| 职位名称 | string |
| 职位描述 | text |
| 职位要求 | text |
| 技能要求 | string[] |
| 学历要求 | string |
| 工作年限 | string |
| 薪资范围 | string |
| 工作地点 | string |

---

### 2.4 AI 匹配模块

#### 匹配流程

```
输入 JD → 解析提取关键信息 → 向量化 → 向量数据库检索 Top-K → LLM 生成匹配理由 → 返回结果
```

#### 向量数据库

- 使用 Milvus 或 Qdrant
- 简历向量 + JD 向量分别存储
- 检索方式：余弦相似度

#### 匹配结果

| 字段 | 说明 |
|------|------|
| 简历 ID | 匹配简历 |
| 匹配分数 | 0-100 分 |
| 匹配理由 | LGM 生成的具体理由 |
| 匹配亮点 | 技能匹配、工作经验匹配等 |

---

### 2.5 Boss 直聘对接模块

#### 功能

| 功能 | 说明 |
|------|------|
| OAuth2 授权 | 企业账号授权接入 |
| 简历同步 | 拉取候选人投递的简历 |
| 同步状态 | 记录同步时间、状态 |

#### 同步策略

- 定时任务：每 6 小时同步一次
- 手动触发：支持立即同步
- 异常处理：失败重试 3 次，记录日志

## 3. 数据流设计

### 3.1 简历上传流程

```
用户上传简历
     │
     ▼
┌─────────────┐
│  文件服务   │  (存储到 MinIO/OSS，返回文件 URL)
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  解析服务   │  (PDF/Word → 纯文本)
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  AI 提取    │  (文本 → 结构化 JSON)
└──────┬──────┘
       │           ┌─────────────────┐
       └──────────▶│  向量数据库     │  (生成向量并存储)
                   └─────────────────┘
                   │
                   ▼
            ┌─────────────┐
            │  PostgreSQL │  (存储结构化数据)
            └─────────────┘
```

### 3.2 AI 匹配流程

```
用户输入 JD（或选择已有 JD）
     │
     ▼
┌─────────────┐
│  JD 解析    │  (提取关键信息)
└──────┬──────┘
       │
       ▼           ┌─────────────────┐
┌─────────────┐    │   向量数据库    │
│  向量化     │───▶│  检索 Top-20   │
└──────┬──────┘    └─────────────────┘
       │                  │
       │                  ▼
       │           ┌─────────────┐
       └──────────▶│  LLM 匹配   │  (生成匹配理由)
                    └──────┬──────┘
                           │
                           ▼
                    ┌─────────────┐
                    │  返回结果   │
                    └─────────────┘
```

## 5. 错误处理

| 场景 | 处理策略 |
|------|----------|
| 文件上传失败 | 重试 3 次，失败返回具体错误 |
| AI 服务不可用 | 降级为纯向量匹配，返回提示 |
| Boss API 异常 | 记录日志，异步重试，展示同步状态 |
| 匹配超时 | 限制匹配简历数量，设置 30s 超时 |

## 7. Git Worktree 开发模式

### 目录结构

```
/home/ubuntu/WS/ai-hiring/
├── main/                    # 主分支
├── worktrees/
│   ├── feature-user/       # 用户模块
│   ├── feature-resume/     # 简历模块
│   ├── feature-jd/        # JD 模块
│   ├── feature-ai/        # AI 匹配模块
│   └── feature-boss/      # Boss 对接模块
```

### 开发流程

1. 从 main 创建功能分支
2. 创建 worktree: `git worktree add ../worktrees/feature-xxx`
3. 在 worktree 中 TDD 开发
4. 测试通过后，合并到 main
5. 删除 worktree

### 模块独立程度

| 模块 | 独立程度 | 说明 |
|------|----------|------|
| 用户模块 | 高 | 可独立开发 |
| 部门模块 | 高 | 依赖用户模块 |
| 简历模块 | 高 | 可独立开发 |
| JD 模块 | 高 | 可独立开发 |
| AI 匹配 | 高 | 依赖简历、JD 模块 |
| Boss 对接 | 中 | 依赖简历模块 |

## 8. CI/CD 流程 (GitHub Actions)

### 触发条件

- push 到 main
- PR 到 main

### 流水线步骤

```
1. 检出代码
2. 安装依赖
3. 运行单元测试
4. 运行集成测试
5. 构建 Docker 镜像
6. 推送镜像到 GitHub Container Registry
```

## 11. 验收标准

1. 用户能够注册、登录、分配角色和部门
2. 用户能够上传简历（PDF/Word），系统自动解析结构化信息
3. 用户能够创建、编辑、删除 JD
4. 用户输入 JD，系统返回匹配简历列表及匹配分数、理由
5. Boss 直聘授权后能够同步简历
6. 部门之间简历可共享，HR 管理员可管理所有部门
7. CI/CD 流水线自动化构建和测试
8. 独立功能通过 Git Worktree 并行开发
